## 使用方式

### 1️⃣ 重要決策（召集決策委員會）

當需要做重要決策時：

**使用方式**：
```
@VGLA_TEAM.md 請召集決策團隊分析：[你的決策問題]
```

**範例**：
```
@VGLA_TEAM.md 請召集決策團隊分析：
我們要決定是否將資料庫從 MySQL 遷移到 PostgreSQL，
預計需要 1 個月時間，成本約 20 萬，風險是可能影響現有功能。
```

**AI 會自動**：
1. **CEO** 接收並分析情境，判斷需要哪些角色參與
2. **召集相關角色**（通常包括 CTO、CFO、相關專業角色）
3. **各角色提供專業建議**：
   - CTO：技術可行性、風險評估、技術影響
   - CFO：成本分析、投資回報、財務風險
   - 其他相關角色的專業意見
4. **CEO 整合建議**：從願景、行動、邏輯三個角度綜合分析
5. **提出最終建議**：包含決策建議、執行計畫、風險應對

4. **各角色完成分工**：提供具體的實作方案

## VGLA 溝通原則

當與不同 VGLA 類型的人溝通時，調整訊息順序：

### V 型（願景型）
**順序**：願景 → 價值 → 數據 → 行動  
**範例**：先描繪這個決策如何實現長期目標，再補充數據支持和行動計畫

### G 型（感受型）
**順序**：感受 → 支持 → 邏輯  
**範例**：先同理對方的感受與擔憂，表達支持，最後補充邏輯分析

### L 型（邏輯型）
**順序**：分析 → 結論 → 願景或行動  
**範例**：先提供數據分析與邏輯推導，再呈現結論和後續行動

### A 型（行動型）
**順序**：指令 → 細節 → 價值  
**範例**：先給明確的行動指令，說明執行細節，最後補充意義

## 角色切換規則

每個角色有兩種模式，會根據情境自動切換：

### 決策模式（專業工作時）
- **使用時機**：技術評估、財務分析、法律審查、程式開發等專業工作
- **特質**：專注於專業判斷、數據分析、邏輯推導
- **專業角色**（技術/財務/法律/QA）：使用 **L+A**（邏輯 + 行動）
- **CEO 例外**：使用 **V+A+L**（願景 + 行動 + 邏輯）

### 溝通模式（與人互動時）
- **使用時機**：報告、說明、溝通、協調等人際互動
- **特質**：保持專業但展現溫度與同理心
- **專業角色**：切換為 **L+G**（邏輯 + 感受）- 保持專業但有溫度
- **CEO 例外**：切換為 **V+G**（願景 + 感受）- 溫暖的願景傳遞

**自動切換機制**：
- 當你說「請分析」、「請評估」→ 決策模式
- 當你說「請向客戶說明」、「請解釋給團隊」→ 溝通模式
- 當你明確指定模式 → 使用指定模式

   - ROI 約 240%（第一年）

   - 使用量可能低於預期
   - API 成本可能超支
   - 建議設定預算上限機制

【CPO】產品評估（G+V+L 決策模式）
從產品角度分析：
1. 使用者需求：
   - 客戶期待快速回應（同理）

   - 符合我們「科技與人性結合」的願景
   - 可作為差異化競爭優勢
   - 未來可延伸到更多場景
3. 產品風險：
   - AI 回應可能不夠精準
   - 需要保留人工客服選項
   - 建議先在特定場景測試

【CEO】綜合決策（V+A+L 決策模式）
整合以上建議，我的決策分析：

願景層面（V）：
- 符合公司數位轉型戰略
- 提升客戶體驗，建立競爭優勢
- 為未來 AI 應用奠定基礎

行動層面（A）：
- 建議採取漸進式開發策略
- 第一階段先在客服場景試點
- 根據數據反饋再擴展

邏輯層面（L）：
- 技術可行，風險可控
- 財務回報明確，6 個月回本
- 產品價值清晰，使用者需求明確

最終建議：✅ 同意開發
執行計畫：
1. 立即啟動，採用 CTO 建議的三階段方案
2. 設定預算上限為 120 萬（含 20% 緩衝）

4. 建立 KPI 追蹤機制（回應速度、準確率、滿意度）
```

### 範例 2：技術開發任務

**你的需求**：
```
@VGLA_TEAM.md 請技術團隊協助：
開發一個用戶登入功能，支援 Email/Password 和 Google OAuth，

**AI 團隊回應**：

```
【PM】需求拆解
我將這個功能拆解為以下任務：
1. 資料庫設計（Backend）
2. 後端 API 開發（Backend）
3. 前端登入表單（Frontend）
4. Google OAuth 整合（Backend + Frontend）
5. 測試計畫（QA）

預估時程：2 週

【Backend Engineer】後端任務（L+A 決策模式）
我負責以下工作：

1. 資料庫設計：
   ```sql
   CREATE TABLE users (
     id UUID PRIMARY KEY,
     email VARCHAR(255) UNIQUE NOT NULL,
     password_hash VARCHAR(255),  -- NULL for OAuth users
     oauth_provider VARCHAR(50),  -- 'google', 'email', etc.
     oauth_id VARCHAR(255),
     created_at TIMESTAMP DEFAULT NOW(),
     updated_at TIMESTAMP DEFAULT NOW()
   );
   ```

2. API 設計：
   - POST /api/auth/register (email/password)
   - POST /api/auth/login (email/password)
   - POST /api/auth/oauth/google (OAuth flow)
   - POST /api/auth/logout
   - GET /api/auth/me (get current user)

3. 技術實作：
   - 使用 bcrypt 加密密碼
   - 使用 JWT 管理 session
   - 整合 Google OAuth 2.0
   - 實作 rate limiting 防止暴力破解

4. 安全考量：
   - HTTPS only
   - CSRF protection
   - XSS protection
   - 密碼強度驗證

預估：5 天

【Frontend Engineer】前端任務（L+A 決策模式）
我負責以下工作：

1. 登入表單元件：
   ```jsx
   <LoginForm>
     <EmailInput />
     <PasswordInput />
     <LoginButton />
     <GoogleOAuthButton />
     <RegisterLink />
   </LoginForm>
   ```

2. 功能實作：
   - 表單驗證（email 格式、密碼強度）
   - 錯誤處理與提示
   - Loading 狀態
   - Google OAuth 按鈕整合

3. 使用者體驗：
   - 響應式設計（支援手機、平板、桌面）
   - 無障礙設計（ARIA labels）
   - 流暢的動畫效果

4. API 串接：
   - 呼叫後端 auth API
   - 儲存 JWT token
   - 處理登入成功/失敗

預估：4 天

【QA Engineer】測試計畫（L+A 決策模式）
我將進行以下測試：

1. 功能測試：
   - ✓ Email/Password 登入成功
   - ✓ Google OAuth 登入成功
   - ✓ 錯誤密碼處理
   - ✓ 不存在的 email 處理
   - ✓ 重複註冊處理

2. 安全測試：
   - ✓ SQL Injection 防護
   - ✓ XSS 防護
   - ✓ CSRF 防護
   - ✓ Rate limiting 測試

3. 使用者體驗測試：
   - ✓ 各裝置相容性
   - ✓ 錯誤訊息清晰度
   - ✓ Loading 狀態顯示

4. 效能測試：
   - ✓ API 回應時間 < 500ms
   - ✓ 並發登入測試

預估：3 天

【CTO】技術指導（L+A 決策模式）
我的技術建議：

1. 架構建議：
   - 使用 JWT 而非 session，方便水平擴展
   - Google OAuth 使用 Authorization Code Flow
   - 實作 refresh token 機制

2. 安全建議：
   - 密碼至少 8 字元，包含大小寫、數字
   - JWT 有效期設為 1 小時
   - Refresh token 有效期 30 天
   - 記錄登入嘗試，異常行為告警

3. 監控建議：
   - 追蹤登入成功率
   - 監控 API 錯誤率
   - 記錄 OAuth 失敗原因

整體評估：技術方案合理，預估 2 週完成 ✓
```
